# Account Constraint Refactor (V20.0.0)

## Goal

Eliminate `Option<>` vault account patterns in Anchor contexts. Move arg validation and defense-in-depth phase checks from handler `require!()` to `#[account]` constraints. Audit-discovered concern (medium): the `torch_vault.as_ref().unwrap()` pattern inside Optional constraint blocks panics at account-validation time, before the handler's `is_some()` check runs.

## Why now

V20.0.0 is a complete breaking change. Constraint-layer validation is strictly more robust than handler-layer — fires earlier, harder to forget, becomes the canonical "what this instruction requires" doc. This is the lowest-cost window to do it.

---

## Decisions

### D-1: Split vault-aware contexts into two non-Optional contexts each

Wallet-funded path keeps the original name (`Buy`, `Sell`, etc.). Vault-routed path uses a `ViaVault` suffix (`BuyViaVault`, `SellViaVault`, etc.). Each context contains exactly the accounts that path needs — no `Option<>`, no `unwrap()` in constraints. Handler code drops the `is_some()` branching; each handler does one thing.

**Affected contexts (10):** `Buy`, `Sell`, `Borrow`, `Repay`, `Liquidate`, `OpenShort`, `CloseShort`, `LiquidateShort`, `StarToken`, `ClaimProtocolRewards`.

**Instruction count: 30 → 40.** Accepted. SDK builders already split (`buildBuyTransaction` vs `buildDirectBuyTransaction`) so TS side maps 1:1.

**Rejected:** keep single instruction with `via_vault: bool` arg + placeholder accounts. Wastes tx bytes, adds CU, doesn't solve the constraint-evaluation issue.

---

### D-2: Defense-in-depth phase constraints

Add constraints that are currently *unreachable* but explicit-is-better-than-implicit. Reader shouldn't have to derive safety from transitive reasoning across handlers.

| Context | Add constraint | Why |
|---|---|---|
| `Liquidate` | `bonding_curve.migrated` + `!bonding_curve.reclaimed` | Loans only exist post-migration; migrated ↔ reclaimed mutually exclusive. Make it explicit. |
| `CloseShort` | same | Shorts only exist post-migration |
| `LiquidateShort` | same | Same as above |
| `Repay` | `bonding_curve.migrated` | Same reasoning |
| `SwapFeesToSol` | `treasury.baseline_initialized` | Kills the pre-V9 bypass branch for new tokens — makes the dead path actually dead |

---

### D-3: Arg validation via `#[instruction(args)]` constraints

Move min/max arg checks from handler `require!()` to context constraints. Validation fires before account loading.

| Args | Constraint to move |
|---|---|
| `BuyArgs` | `args.sol_amount >= MIN_SOL_AMOUNT` |
| `OpenShortArgs` | `args.sol_collateral > 0 \|\| args.tokens_to_borrow >= MIN_SHORT_TOKENS` |
| `BorrowArgs` | `args.collateral_amount > 0 \|\| args.sol_to_borrow >= MIN_BORROW_AMOUNT` |
| `SellArgs` | `args.token_amount > 0` |

Handler `require!()` removed for these; constraint becomes the single source.

---

### D-4: Out of scope (stays in handler)

- **Pool depth checks** (`pool_sol >= MIN_POOL_SOL_LENDING`): requires `Rent::get()?` + DeepPool reserve read. Not constraint-expressible.
- **LTV computation, interest accrual, bad-debt math:** state mutation, not validation.
- **Cross-account reads** (DeepPool reserves, treasury vault balances): can't be constraint.
- **Sysvar-dependent checks** (slot/timestamp gates in revival/reclaim): stay in handler.

---

### D-5: Out of scope for this refactor (broader)

- Schema changes to `#[account]` types (none needed)
- New features or instruction additions beyond the splits
- Math/handler logic changes
- Anything not strictly about constraint placement

---

## Implementation plan

1. **Types (contexts.rs):** add `BuyViaVault`, `SellViaVault`, `BorrowViaVault`, `RepayViaVault`, `LiquidateViaVault`, `OpenShortViaVault`, `CloseShortViaVault`, `LiquidateShortViaVault`, `StarTokenViaVault`, `ClaimProtocolRewardsViaVault`. Original contexts shrink (no Optional vault fields). Add defense-in-depth + arg constraints per D-2 / D-3.
2. **lib.rs:** add 10 new instruction entry points (`buy_via_vault`, etc.).
3. **Handlers:** split each vault-aware handler into wallet-only + vault-only variants. Each variant shorter and clearer than the original Optional-branching version. Extract shared logic to private helpers in the same module.
4. **SDK (packages/sdk):** update `buildBuyTransaction` to call `buy_via_vault`, `buildDirectBuyTransaction` to call `buy`. Same for all 10. Re-generate types from new IDL.
5. **Tests:** update e2e tests to use new instruction names where they exercise vault paths. No new tests required for this refactor — Kani/proptest coverage of math is unchanged.

---

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| Instruction count grows (30 → 40) | Each instruction is simpler; SDK already splits the call sites |
| Compute budget per call | No change — same work, just routed to a smaller context |
| IDL regeneration required | Already a V20 breaking change; SDK refresh is part of V20.0.0 anyway |
| Handler code duplication | Mitigate via shared private helpers (e.g., `apply_buy_curve_math(..)`) called from both wallet and vault paths |
| Anchor `init_if_needed` semantics differ between split contexts | Verify per-context; `init_if_needed` on accounts like `user_position` stays in both, `init_if_needed` on `buyer_token_account` only in wallet path |

## Acceptance criteria

- [ ] 0 occurrences of `torch_vault.as_ref().unwrap()` or `.unwrap()` inside any `#[account]` block

- [ ] `anchor build` succeeds; IDL regenerates cleanly
- [ ] SDK builds without type errors against the new IDL
